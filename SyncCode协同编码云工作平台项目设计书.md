# 一、项目概述

## 1.1 项目背景

Vibe Coding 开发模式注重自由度与创意快速落地，但其在团队协作场景中存在显著的结构性缺陷，主要体现在以下方面：

- 开发标准缺失，导致系统可维护性不足；
- 协作链路断裂，造成需求与代码开发脱节；
- 工程化能力缺位，难以实现项目规模化落地。

目前市场上主流工具存在功能局限，无法形成完整协作闭环，具体如下：

- GitHub：代码管理能力较强，但实时协作功能薄弱；
- Notion：协作功能完善，但缺乏工程化管控能力；
- Replit：云开发支持较好，但缺乏统一的开发规范约束。

上述工具均未能有效解决“创意、工程、协作、质量”的全流程闭环问题。

## 1.2 项目核心目标

本项目旨在构建一个具备多维度能力的统一协同编码平台，核心能力涵盖：

> 文档级实时协作、代码级工程规范、环境级沙箱隔离、质量级自动评价

通过平台建设，实现两大核心转变：

- 从“自由开发”向“可控化工程开发”转变；
- 从“个人效率优先”向“团队协同效率提升”转变。

## 1.3 产品定位

SyncCode 产品定位为：

> **文档式协作平台、云开发环境与工程质量控制系统的一体化解决方案**

与同类产品相比，其差异化优势如下表所示：

| 维度     | SyncCode                             |
| -------- | ------------------------------------ |
| 协作模型 | 文档式实时协作与代码协作深度融合     |
| 开发环境 | 基于容器技术的沙箱隔离环境           |
| 规范控制 | 开发规范强制执行，保障代码一致性     |
| 质量评估 | 代码质量自动量化评价，实现全流程管控 |
| 工作模式 | 创意构思与代码开发双模式协同运行     |

# 二、系统整体架构

## 2.1 架构风格

本系统采用云原生架构设计，结合微服务与事件驱动架构（EDA），引入CRDT实时协作模型，确保系统的高可用性、可扩展性与实时性，具体架构风格如下：

- 云原生架构（Cloud Native）：适配云端部署，支持弹性伸缩；
- 微服务 + 事件驱动（EDA）：实现业务解耦，提升系统响应效率；
- CRDT 实时协作模型：保障多人实时编辑的一致性与可靠性。

> CRDT（Conflict-free Replicated Data Type，无冲突复制数据类型）是一种用于分布式系统中实现**无冲突实时协作**的数据结构，核心目标是解决多人同时编辑、分布式部署场景下的数据一致性问题，无需中央服务器协调即可实现最终一致性。


# 三、核心数据模型设计

## 3.1 核心实体

系统核心实体包括工作区、文档、任务、沙箱与评价，各实体数据结构定义如下：

### Workspace（工作区）

```sql
CREATE TABLE workspace (
  id BIGINT PRIMARY KEY,
  name VARCHAR(255),
  owner_id BIGINT,
  mode ENUM('idea','code'),
  created_at DATETIME
);
```

### Document（文档，基于CRDT）

```sql
CREATE TABLE document (
  id BIGINT PRIMARY KEY,
  workspace_id BIGINT,
  type VARCHAR(50),
  content JSON, -- 存储Yjs协作结构数据
  version INT
);
```

### Task（任务）

```sql
CREATE TABLE task (
  id BIGINT PRIMARY KEY,
  workspace_id BIGINT,
  assignee_id BIGINT,
  status VARCHAR(50),
  branch_name VARCHAR(100)
);
```

### Sandbox（沙箱）

```sql
CREATE TABLE sandbox (
  id BIGINT PRIMARY KEY,
  user_id BIGINT,
  workspace_id BIGINT,
  container_id VARCHAR(255),
  branch_name VARCHAR(100),
  status VARCHAR(50)
);
```

### Evaluation（评价）

```sql
CREATE TABLE evaluation (
  id BIGINT PRIMARY KEY,
  commit_id VARCHAR(100),
  score FLOAT,
  detail JSON
);
```

# 四、核心技术机制

## 4.1 实时协作一致性模型

### 模型设计

为保障多人实时协作的一致性与可靠性，系统采用分层协作模型，具体设计如下：

- 文档层：采用CRDT（Yjs）模型，实现去中心化的实时协作与最终一致性；
- 代码层：基于Git版本控制，实现代码的版本管理与分支协作。

### 同步流程

文档与代码的同步流程如下：

![mermaid-1776253499304](https://files.seeusercontent.com/2026/04/15/z6sQ/mermaid-1776253499304.png)

### 冲突策略

针对不同层级的冲突，系统采用差异化解决策略，具体如下表所示：

| 冲突类型 | 解决方式                                              |
| -------- | ----------------------------------------------------- |
| 文档冲突 | 通过CRDT模型自动合并，无需人工干预                    |
| 代码冲突 | 基于Git机制，结合可视化差异对比工具，支持人工协同解决 |

为实现可工程化的实时协作能力，系统基于RGA（Replicated Growable Array）模型构建CRDT数据结构。

### 4.1.1 数据节点结构（Node）

```
每个文档字符或元素定义为一个节点：

Node = {
  id: (client_id, counter),
  value: 数据内容,
  deleted: 是否删除,
  next: 指向下一个节点
}
```

------

### 4.1.2 操作模型（Operation）

```
操作定义如下：

Op = {
  id: 唯一标识,
  type: Insert | Delete,
  target: 目标位置,
  value: 插入内容,
  timestamp: 时间戳,
  actor: 操作者ID
}
```

------

### 4.1.3 合并规则

```
系统通过以下规则解决并发冲突：

当多个操作作用于同一位置时：
- 按 timestamp 排序；
- 若时间相同，则按 client_id 排序；

最终保证所有副本状态一致（最终一致性）。
```

------

### 4.1.4 快照机制

```
为避免CRDT操作日志无限增长，系统引入快照机制：

- 每执行N次操作（默认1000次），生成一次快照；
- 将历史操作压缩为当前状态；
- 后续同步基于快照 + 增量操作。

该机制有效控制内存与网络开销。
```

## 4.2 双模式引擎

### 创意模式与编码模式转换机制

系统支持创意模式与编码模式的无缝切换，创意模式产出可自动转换为编码开发基准，具体转换内容包括：

- API接口定义文档；
- 核心数据结构草案；
- 初始开发任务列表。

该转换机制本质为轻量级文档驱动开发（DDD），确保创意构思与代码开发的一致性。

## 4.3 沙箱环境调度

### 技术栈选型

沙箱环境基于容器技术构建，选用以下核心技术栈，保障环境隔离性与轻量化：

- 容器运行时：Containerd；
- 容器编排：Kubernetes。

### 调度策略

为优化资源利用效率，降低冷启动时间，沙箱环境采用以下调度策略：

#### 1. 预热池机制（Warm Pool）

系统维护常驻基础镜像容器池，当用户申请沙箱时，可快速复用容器资源，将冷启动时间控制在2秒以内。

#### 2. 资源分级配置

根据开发任务复杂度，沙箱资源分为三个等级，具体配置如下表所示：

![image-20260415194539666](https://files.seeusercontent.com/2026/04/15/8Kti/image-20260415194539666.png)

#### 3. 资源回收机制

为避免资源浪费，沙箱环境采用自动回收策略：

- 沙箱闲置10分钟后，自动进入休眠状态，释放部分资源；
- 沙箱闲置24小时后，系统自动销毁该沙箱，彻底释放所有资源。

## 4.4 规范驱动引擎

系统内置规范驱动引擎，实现开发规范的自动化落地与管控，具体支持能力如下：

- 规范校验工具：支持ESLint、CheckStyle等主流规范校验工具；
- 代码分析能力：支持AST（抽象语法树）分析，实现细粒度规范校验。

规范校验分为两个核心执行阶段，确保规范落地的全面性：

1. 编码实时校验：在用户编码过程中，实时提示违规内容，引导用户规范开发；
2. 提交前强制校验：代码提交前，系统执行全量规范校验，违规代码禁止提交。

## 4.5 SDD开发机制

### SDD（Software Development Document）开发定义

SDD开发机制是系统工程化管控的核心补充，聚焦软件开发文档的标准化、规范化生成与迭代，实现“文档与代码同步、规范与开发同行”，确保开发过程可追溯、可复用、可维护，适配团队协作与项目规模化落地需求。

### 核心功能与实现方式

- 标准化模板生成：内置SDD标准模板，涵盖需求说明、接口设计、数据模型、测试用例等核心模块，支持根据项目类型（如后端服务、前端应用）自定义模板，减少文档编写冗余工作；
- 代码与文档联动：基于代码注释、接口定义自动生成SDD相关章节，当代码发生变更时，同步提醒用户更新对应文档内容，确保文档与代码的一致性，避免“文档与实现脱节”问题；
- 版本化管理：依托Git版本控制机制，实现SDD文档的版本追踪与回溯，支持多人协同编辑文档，结合CRDT实时协作模型，避免文档编辑冲突，留存每一次修改记录；
- 规范校验集成：将SDD文档规范纳入系统规范驱动引擎，对文档的完整性、规范性、一致性进行自动校验，未达标的文档禁止提交，确保文档质量。

![image-20260415194611166](https://files.seeusercontent.com/2026/04/15/0Boh/image-20260415194611166.png)

### 与系统核心模块的联动

SDD开发机制深度联动文档服务、规范驱动引擎与自动化评价系统：文档服务提供SDD实时协作与版本管理支撑，规范驱动引擎实现SDD规范校验，自动化评价系统将SDD文档完整性纳入D（文档一致性）指标评分，形成“文档生成-协作-校验-评价”的闭环管理。

## 4.6 Harness Agent 集成

### Harness Agent 定位与作用

Harness Agent作为系统持续集成/持续部署（CI/CD）能力的核心载体，部署于K8s沙箱集群节点，负责衔接沙箱环境、微服务集群与部署架构，实现开发、测试、部署全流程的自动化调度与管控，提升系统部署效率与稳定性。

### 核心能力与部署方式

- 自动化部署调度：接收系统部署指令，实现微服务、沙箱环境的自动化部署、升级与回滚，支持按环境（开发、测试、生产）区分部署策略，适配不同阶段的部署需求；
- 环境监控与反馈：实时采集沙箱环境、微服务运行状态数据（如CPU、内存占用、服务可用性），同步反馈至系统监控模块，当出现异常时触发告警，助力问题快速排查；
- CI/CD流程衔接：联动Git服务与自动化评价系统，当代码提交并通过规范校验、评分达标后，Harness Agent自动触发构建、测试、部署流程，实现“代码提交-校验-部署”的全自动化闭环；
- 轻量化部署：采用容器化部署方式，与系统K8s沙箱集群深度适配，占用资源低，支持弹性伸缩，可根据部署任务量自动调整Agent实例数量，优化资源利用效率。

Harness Agent与沙箱服务、部署架构、自动化评价系统协同工作：沙箱服务为Agent提供部署目标环境，部署架构为Agent提供拓扑支撑，自动化评价系统为Agent触发部署流程提供评分依据，三者协同实现系统开发部署全流程的自动化、可控化。

## 4.7 自动化评价系统

系统构建自动化软件工程评价体系，通过量化指标对代码质量与协作效果进行客观评价，核心评分函数如下：

$$Score = 0.4C + 0.3Q + 0.2S + 0.1D$$

### 子指标定义

评分函数中各子指标的定义与计算方式如下：

- C（规范合规度）：以代码规范校验通过率为核心计算依据；
- Q（代码质量）：综合代码圈复杂度、代码重复率、Code Smell数量等指标计算；
- S（协作规范）：结合代码提交频率、任务按时完成率等协作行为指标计算；
- D（文档一致性）：以API文档与代码实现的一致性为核心计算依据。

评价系统可集成SonarQube等第三方代码质量检测工具，丰富评价维度，提升评价准确性。

各指标权重设计如下：

- C（规范合规度）：40%，保障代码统一性；
- Q（代码质量）：30%，反映代码复杂度与可维护性；
- S（协作规范）：20%，体现团队协作效率；
- D（文档一致性）：10%，确保文档与代码同步。

为避免单次提交对评分影响过大，系统引入时间衰减模型：$$Score_t = λ × 当前评分 + (1 - λ) × 历史评分$$，其中 λ 默认取 0.7，用于平衡短期表现与长期稳定性。

## 10.2 分布式高可用设计

------

### 10.2.1 数据层高可用

```
系统数据层采用分布式与高可用架构设计：

MySQL：
- 主从复制架构（Primary + Replica）；
- 支持读写分离；
- 主节点负责写操作，从节点负责读操作；
- 故障时自动切换主节点。

Redis：
- 采用Redis Cluster模式（多主多从）；
- 支持数据分片与高可用；
- 用于缓存、会话管理与CRDT状态同步。
```

------

### 10.2.2 WebSocket 分布式协作

```
由于实时协作依赖WebSocket长连接，而WebSocket具有状态性，系统采用以下分布式方案：

1. 多节点部署Yjs协作服务；
2. 引入Redis Pub/Sub实现节点间状态同步；

同步流程如下：

客户端连接任意节点
↓
节点接收CRDT更新
↓
通过Redis Pub/Sub广播
↓
其他节点同步状态

该机制确保多节点环境下文档实时一致性。
```

------

### 10.3.3 服务高可用策略

```
所有微服务采用Kubernetes Deployment部署，配置如下：

- 每个服务至少2个副本（replicas ≥ 2）；
- 使用Liveness Probe与Readiness Probe进行健康检查；
- Pod异常自动重启；
- 节点故障自动迁移。

系统整体实现无单点故障（SPOF）。
```

# 十二、风险补充

系统在运行过程中可能面临以下潜在风险，对应的解决方案如下：

### 1. CRDT性能问题

风险描述：随着文档编辑操作增多，CRDT操作日志可能出现膨胀，影响系统性能。

解决方案：采用文档分片存储与局部同步机制，减少全局同步的数据量，优化CRDT性能。

### 2. 沙箱成本过高

风险描述：大量沙箱环境同时运行，会导致服务器资源占用过高，增加部署成本。

解决方案：采用按需调度与自动回收机制，根据用户实际使用需求分配资源，闲置沙箱及时释放资源，降低运行成本。

### 3. SDD文档一致性风险

风险描述：代码变更后，用户未及时更新SDD文档，导致文档与代码脱节，影响开发协作与项目追溯。

解决方案：设置文档与代码联动告警机制，代码变更后自动提醒用户更新对应SDD章节；同时在代码提交校验阶段，增加SDD文档一致性校验，未同步更新的代码禁止提交。

### 4. Harness Agent稳定性风险

风险描述：Agent部署节点故障或通信异常，导致自动化部署、监控功能失效，影响CI/CD闭环流程。

解决方案：采用Agent集群部署模式，单个节点故障时自动切换至备用节点；引入心跳检测机制，实时监控Agent运行状态，异常时触发告警并自动重启Agent，保障服务连续性。

# 十三、项目总结

SyncCode 平台的核心定位并非单纯的代码管理平台，而是一套完整的工程协作系统，其核心特征为：

> **以文档为入口、以规范为约束、以沙箱为执行、以评分为闭环的工程协作系统**

该平台核心解决两大核心问题：

- 如何将创意构思转化为可协作、可落地的软件工程；
- 如何在保障开发自由度的同时，确保工程质量与规范落地。

SDD开发机制与Harness Agent的集成，进一步完善了系统工程化能力：SDD机制实现了开发文档的标准化、协同化管理，解决了“文档与代码脱节”的行业痛点；Harness Agent则打通了开发、测试、部署的全流程自动化，提升了系统部署效率与稳定性，二者协同推动系统从“协作管控”向“全流程工程化”升级，强化了平台的差异化竞争优势。
